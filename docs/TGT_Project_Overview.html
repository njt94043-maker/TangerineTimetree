1. What Is This?

The Green Tangerine is a 4-piece live music band based in the UK. Nathan Thomas (drums) is a sole trader who manages the business side — booking gigs, invoicing clients, and splitting payments between the four members.

This project is a monorepo containing four applications (plus one planned) that together handle everything the band needs: from booking a wedding gig and sending a professional invoice, to running a click track in Nathan's in-ear monitors on stage, to capturing YouTube audio for practice material.

All apps share one dark neumorphic visual identity — the same colours, fonts, and design language. They share one Supabase cloud database so that a song added on one platform appears on all others instantly. They are one family of tools, not four separate products.

The stage performance app. Built natively because live click tracks need sub-millisecond audio latency that web browsers cannot achieve. The C++ audio engine runs in the Oboe audio callback — the same thread that feeds the speaker hardware.

Why it exists: Nathan wears in-ear monitors on stage. He needs a rock-solid click track synced to each song's tempo map, with stem playback for practice. Web Audio API has 5-20ms latency — fine for practice, not for a live stage click in your ears.

4. How Data Flows

The Song Pipeline

Getting a song from "I found this on YouTube" to "the click track is in my ears on stage" involves multiple steps across multiple apps:

1
Capture — Nathan records audio from YouTube (via Chrome extension or WASAPI system loopback). Capture analyses BPM and key using librosa, encodes to high-quality MP3 via FFmpeg, and stores it locally with metadata.
2
Import — From the web app, Nathan browses his Capture library and imports a track. The web app creates a Song record in Supabase, uploads the MP3 to cloud storage, and maps Capture's metadata (title, artist, BPM, category) to the Song schema.
3
Process — The web app triggers Google Cloud Run, which runs madmom (neural network beat detection) to generate a precise beat map — an array of timestamps for every beat in the song. Optionally, Demucs splits the audio into stems (drums, bass, vocals, other).
4
Play — The beat map and stems are stored in Supabase. Both the web app and Android app read them. When Nathan hits play, the click scheduler fires oscillator pulses at exactly the right moments, synced to the beat map. Stems load into the mixer for isolated practice.
5
Perform — On stage, Nathan loads a setlist. The Android app's C++ engine plays the click in his in-ear monitors with sub-millisecond accuracy. The web app does the same for practice sessions at a slightly higher latency.

The Business Pipeline

1
Booking — A venue or client contacts the band. Any member opens the calendar, taps "Add Gig", and fills in the booking wizard (venue, client, date, fee, event type, timings).
2
Quote — For client gigs (weddings, corporate), a formal quote is generated from one of 7 professional PDF templates and shared with the client.
3
Invoice — After the gig, an invoice is generated (7 styles + 3 seasonal themes). The fee is automatically split four ways.
4
Payment — When marked paid, three receipts are generated (one per non-Nathan member — Nathan pays himself, so no self-receipt).

5. Web App — Every Feature and Why

Calendar & Booking

The band's shared diary. Every member needs to know when and where gigs are.

  • Calendar Month view with coloured day dots — green for gigs, purple for practice, red for member-away days. Swipe to change months. Tap a day to see details.
  • Day Detail Shows all events for a day. "Add Gig", "Add Practice", and "I'm Away" buttons. Displays gig status (enquiry/confirmed), venue, fee, and an INCOMPLETE badge if data is missing.
  • Booking Wizard Two flows: Quick Entry (pub gigs — minimal fields) and Full Booking (weddings/corporate — venue, client, fee, deposit, event type, timings, load-in details). Intentionally comprehensive — it trains members to capture everything Nathan needs for invoicing.
  • Away Dates Any member can mark themselves unavailable. If one member is away, the band is unavailable — this shows as a red day on everyone's calendar.
  • Change Summary On app open, a banner shows what changed since you last looked — gigs added, dates changed, members marked away. Prevents "I didn't know about that gig" problems.

Invoicing, Quotes & Receipts

Nathan is a sole trader. Professional invoicing is not optional — it's how the band gets paid.

  • Invoice Wizard 3-step flow: select client/venue, enter gig details, preview PDF. Seven visual styles (Classic, Premium, Clean, Bold, Elegant, Minimal, Modern) plus three seasonal themes (Christmas, Halloween, Valentine). Full A4 preview before generating.
  • Invoice List Searchable, filterable list of all invoices. Filter by status (draft/sent/paid), date range. CSV export for accounting.
  • Quote Builder Same quality as invoices but for pre-gig pricing. Sent to clients/venues to secure bookings. Seven matching visual styles.
  • Formal Invoices A more structured invoice format for corporate clients who need specific formatting (PO numbers, VAT details).
  • Receipt Split When an invoice is marked paid, three receipts are automatically generated — one for each non-Nathan member. The fee divides by 4, first receipt gets any rounding remainder. Nathan doesn't get a receipt because he pays himself.
  • Bill-To Resolution Invoices can target a venue (pub pays directly) or a client (agency books the venue). The PDF template automatically pulls the correct name, address, and contact details.
  • PDF Sharing Adam shares invoices with venues via WhatsApp/messenger on Nathan's behalf. The app generates the PDF so Adam just shares a file — no filling in details on the phone.

Library — Songs & Setlists

The launchpad for all music. Browse, search, and tap to enter Live, Practice, or View mode.

  • Song Library All songs the band has added. Four categories: TGT Cover, TGT Original, Personal Cover, Personal Original. Filter by scope (All / TGT / My Songs / Shared With Me) and type (Covers / Originals). Search by title or artist.
  • Song Cards Raised neumorphic cards showing title, artist, BPM, key, and category badges. Tap to expand and reveal Live / Practice / View launch buttons.
  • Setlist Library Ordered collections of songs for a gig. Types: TGT setlists and Other Band setlists (for dep gigs where Nathan sits in with another band).
  • Song Form Edit metadata: name, artist, BPM, time signature, key, category, lyrics, chords, notes. Nathan (drums role) sees additional metronome settings: subdivision, swing, accent patterns, click sound, count-in bars. Upload practice tracks, manage stems, view/manage recorded takes.
  • Import from Capture Browse Nathan's local Capture library, select a track, map its metadata to a Song, upload the MP3 to Supabase, and trigger Cloud Run processing for beat maps and stems.
  • Song Sharing Personal originals (works in progress) are private by default. The owner can share with specific members. Personal covers are visible to all — "good to know what everybody already knows."

Player — Live, Practice & View Modes

The performance engine. Three modes in one player shell — because a song is a song whether you're on stage, in the practice room, or reviewing a recording.

  • Live Mode For gigs. Click track + visual beat display + lyrics/chords reference. No backing tracks — just the click in your ears and the info on screen. Fullscreen, no navigation drawer. Queue = the setlist for this gig.
  • Practice Mode For rehearsal. Everything Live Mode has, plus: backing track playback, stem mixer (isolate drums, bass, vocals, other), pitch-preserved speed control (slow down to learn, speed up to challenge), A-B section looping, waveform display.
  • View Mode For reviewing recordings. Plays back your recorded takes with all stems. If you have a local video recording, it shows in the hero area. If not, a visualiser fills the space.
  • Click Track A metronome that follows the song's beat map (generated by madmom from the audio). Every click sounds identical — no accent on beat 1, no downbeat distinction. Quick attack, slow release on visuals. If no beat map exists, click uses manual BPM.
  • Stem Mixer Per-stem gain faders with mute and solo. Lives in the bottom drawer — not permanently visible. Keeps the main screen clean for the visual hero.
  • Visual Hero Three visualiser modes (Spectrum, Rings, Burst) that flash in time with the click. Beat-synced, not audio-reactive. Card-level beat glow is the default. Fullscreen edge glow is an experimental opt-in toggle.
  • Recording Record a take over the backing track. Camera optional (video stays local, audio uploads as a stem). Count-in, overdub mix via drawer, post-recording options: discard, save, save & preview, save & re-take.
  • Takes Multiple takes per user per song. One "best take" per user promotes to the shared stem list — all members can hear it in their mixer. Non-best takes stay local.
  • Queue The list of songs from wherever you launched. Tap a song from a setlist? That setlist is your queue. Tap from All Songs? All songs is your queue. Previous/Next navigate within the queue. You can switch to a different list from the queue overlay.
  • Display Toggles In the bottom drawer: click, flash, lyrics, chords, notes, drums, visualiser — each member configures their own experience. Per-user preferences saved to the database.

Other Web Features

  • Dashboard Stats overview: gigs this month/year, income totals, payment status breakdown. Tax year filtering. Quick glance at band activity.
  • Settings User profile, band settings, bank details (for invoices — not hardcoded), public site configuration (service catalogue, reviews, gallery, background images).
  • Public Site The band's website at thegreentangerine.com. Photo gallery, service descriptions, reviews, upcoming public gigs, booking contact form. Built into the same Vite app — no separate deployment.
  • Stage Prompter Lyrics and chords display for a tablet on a music stand. Navigate through a setlist. Fullscreen dark mode. No auto-scroll — musicians glance at it between sections.
  • GigHub Pipeline view for a gig: from booking to quote to invoice to payment. One screen to see where everything stands.
  • Offline Support Mutations queue locally when offline, replay on reconnect. Calendar data cached. Not full offline-first, but graceful degradation.
  • PWA Installable on phone and desktop. Service worker with auto-update (skipWaiting + clientsClaim + 5-minute poll for new versions).

6. Android App (GigBooks) — Every Feature

The native stage companion. Same features as the web player, but with a C++ audio engine for sub-millisecond click timing.

  • Calendar Same as web — month view with coloured dots, day detail panel, gig info. Reads from the same Supabase tables.
  • Library Songs + Setlists tabs with filter dropdowns (Scope + Type), search, raised NeuCard song cards. Tap to expand, launch into Live / Practice / View.
  • Live Mode C++ click track via Oboe (sub-ms latency). Visual beat display, lyrics/chords, queue navigation. Fullscreen with bottom drawer for mixer and display toggles.
  • Practice Mode Same C++ engine plus: SoundTouch pitch-preserved speed control, waveform strip with A-B loop, stem mixer, recording with post-recording options.
  • View Mode Take playback with stem mixer. Reuses Practice infrastructure.
  • Song Form Full metadata editing, practice track management, beat analysis trigger.
  • Settings Player preferences (click sound selection).
  • Navigation Compose ModalNavigationDrawer with 3 items: Calendar, Library, Settings. Player is persistent — survives navigation.

The Android app does NOT handle invoicing, quotes, PDF, clients, or venues. That stays on the web app.

7. Capture Tool — Every Feature

The entry point for all audio in the ecosystem. Nathan's local tool for grabbing practice material.

  • WASAPI Capture Records system audio (anything playing through speakers/headphones) via Windows Audio Session API loopback. Armed mode waits for audio to start, then records automatically.
  • Chrome Extension Side panel in Chrome for capturing audio from specific browser tabs. WebSocket connection to the backend. Useful for YouTube capture without grabbing other system sounds.
  • BPM & Key Detection librosa analyses the captured audio to detect tempo and musical key. Auto-populates metadata so Nathan doesn't have to enter it manually.
  • MP3 Encoding FFmpeg encodes captured WAV/WebM to high-quality MP3 (LAME V0). ID3 tags embedded with metadata.
  • Waveform Generation 800-point amplitude data + PNG thumbnail for visual display in the UI.
  • Track Library Browse, search, favourite, and manage all captured tracks. Edit metadata (title, artist, tags). Local SQLite database.
  • Import Server Serves tracks via REST API (localhost:9123) for the web app's import pipeline to consume.

Google Cloud Run

The beat-analysis service runs madmom (recurrent neural network beat detection) and Demucs (AI source separation) on uploaded tracks. Triggered asynchronously via Cloud Tasks after a practice track upload. Cold start ~90 seconds, hidden by the async pattern — user sees a "processing" indicator and can continue working.

Why Cloud Run? madmom is Python + numpy — can't run in Supabase Edge Functions (Deno/JS only). Google's free tier (2M requests/month) covers band scale at zero cost.

11. Honest Assessment

What's Working

The business side of this project is solid. Calendar, booking, invoicing, quotes, PDF generation, public website — these features are production-ready. The whole band could start using them daily right now. The Supabase backend is stable, the data model is comprehensive (26 tables with proper relationships), and TypeScript compiles clean.

The Android app's C++ audio engine works reliably — click is tight, stems play, mixer controls the right channels.

What's Not Working

The web click track has consumed 9 sprint sessions (S51-S59) in a fix/break/revert cycle. The root issues have been architectural: AnalyserNode wired in series (killed audio), rAF vs setInterval race conditions, scheduler gated behind a DB preference that got stuck at "off", foreground vs background AudioContext differences on mobile.

Each session has fixed one thing and broken another because there is no stored reference implementation to compare against. Research has been done (Chris Wilson's scheduling pattern, Tone.js source) but only the conclusions were recorded, not the full patterns. When a new symptom appears, the AI falls back to guessing because the underlying knowledge isn't stored anywhere retrievable.

The documentation system is part of the problem. Ten SOT files totalling ~5,000 lines consume a large portion of the AI's context window at session start. Information about the click is scattered across 5+ files. SESSION_LOG.md is 1,778 lines and has never been pruned. SPRINT_PROMPTS.md hasn't been updated since S45. The pain journal — designed specifically to prevent repeat failures — has only one entry.

What Needs to Change

  • Fix the click once, properly. Check the database preference. Store a complete reference implementation document. No more guessing.
  • Remove debug UI from production. The debug banner and test beep should not be on the live website.
  • Prune the docs. Archive SESSION_LOG to last 5 sessions. Kill SPRINT_PROMPTS. Consolidate pain_journal into gotchas. Mark superseded decisions. This alone would cut AI context load by 40%.
  • Store research properly. A "Web Audio Reference" document with the complete Chris Wilson pattern, AudioContext lifecycle rules, and OscillatorNode usage spec. Written once, used forever.
  • Stop adding features until what exists is stable. The business half is ready for daily use. The audio half needs stabilisation, not more features.

The Priority

Get the web click track working reliably. Everything else — visualisers, recording, import polish, ClickTrack — is downstream of that. A band tool that can't keep time is not a band tool.
